💡 Consejos Clave para el Examen de SQL

  • Diferencia DDL vs. DML: DDL (CREATE, ALTER, DROP) define la estructura de los datos (la caja). DML (SELECT, INSERT, UPDATE, DELETE) manipula los datos dentro de esa estructura (el contenido).
  • Claves Primarias vs. Foráneas: La PRIMARY KEY identifica de forma única una fila. La FOREIGN KEY (con REFERENCES) asegura la integridad referencial obligando a que el valor exista en la tabla padre.
  • Orden Lógico de Ejecución de SELECT: FROM & JOIN ➡️ WHERE ➡️ GROUP BY ➡️ HAVING ➡️ SELECT ➡️ ORDER BY ➡️ LIMIT. Nota: No puedes usar un alias definido en el SELECT dentro del WHERE porque el WHERE se ejecuta antes.
  • WHERE vs. HAVING: WHERE filtra registros individuales antes de agrupar. HAVING filtra grupos enteros después de aplicar GROUP BY y funciones de agregación (COUNT, SUM, etc.).
  • Funciones de Ventana (OVER): A diferencia de GROUP BY que colapsa filas, OVER calcula subtotales o acumulados fila por fila, permitiendo análisis avanzados sin perder el detalle del registro.

Documentación Técnica del Proyecto SQL y Análisis de Esquema

1. Resumen Ejecutivo

Este documento proporciona una documentación técnica exhaustiva del Sistema de Gestión de Coworking (denominado bajo el nombre clave 'Tienda de disfraces'). La base de datos está diseñada en MySQL 8.0+ y normalizada en Tercera Forma Normal (3FN). Contiene una fuerte lógica de automatización en base de datos mediante triggers, procedimientos almacenados y eventos que garantizan el correcto funcionamiento del negocio de forma autónoma.

Estadísticas de Objetos en el Esquema:

  • Tablas Estructurales: 31 tablas relacionales con índices y llaves foráneas correspondientes.
  • Vistas de Seguridad (RLS): 4 vistas para el control de privacidad de datos.
  • Triggers Activos: 13 disparadores automáticos de auditoría y validación.
  • Procedimientos Almacenados: 20 stored procedures para lógica compleja.
  • Funciones de Usuario: 20 stored functions de cálculo utilitario.
  • Eventos Programados: 20 scheduled events de mantenimiento diario/semanal/mensual.
  • Roles de Seguridad: 5 roles lógicos predefinidos.
  • Consultas Predefinidas: 100 consultas SQL documentadas.

Procedimiento: RegistrarNuevaMembresia

Propósito: Registra la asignación de una nueva membresía validando previamente que el usuario no tenga otra membresía activa y calculando automáticamente la fecha de expiración.

DELIMITER //
CREATE PROCEDURE RegistrarNuevaMembresia (
    IN p_usuario_id VARCHAR(36),
    IN p_tipo_membresia_id VARCHAR(36),
    IN p_fecha_inicio DATE
)
BEGIN
    DECLARE v_duracion INT;
    DECLARE v_precio DECIMAL(10,2);
    DECLARE v_fecha_fin DATE;
    DECLARE v_membresia_activa INT;

    SELECT COUNT(*) INTO v_membresia_activa FROM membresia_usuario 
    WHERE usuario_id = p_usuario_id AND estado = 'ACTIVA';
    
    IF v_membresia_activa > 0 THEN 
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'Error: El usuario ya tiene una membresia activa.';
    END IF;

    SELECT duracion_dias, precio_base INTO v_duracion, v_precio FROM tipos_membresia 
    WHERE id = p_tipo_membresia_id AND estado = 'ACTIVO';
    
    IF v_duracion IS NULL THEN 
        SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'Error: El tipo de membresia no existe o esta inactivo.';
    END IF;

    SET v_fecha_fin = DATE_ADD(p_fecha_inicio, INTERVAL v_duracion DAY);

    INSERT INTO membresia_usuario(
        usuario_id, tipo_membresia_id, fecha_inicio, fecha_fin, estado, precio_pagado
    ) VALUES (
        p_usuario_id, p_tipo_membresia_id, p_fecha_inicio, v_fecha_fin, 'ACTIVA', v_precio
    );
END //
DELIMITER ;

Propósito: Calcula y retorna la cantidad de días restantes antes de que venza la membresía activa del usuario.

CREATE FUNCTION fn_dias_restantes_membresia(p_usuario_id VARCHAR(36)) 
RETURNS INT 
DETERMINISTIC READS SQL DATA
BEGIN
    DECLARE v_fecha_fin DATE;
    DECLARE v_dias INT;
    
    SELECT MAX(fecha_fin) INTO v_fecha_fin FROM membresia_usuario 
    WHERE usuario_id = p_usuario_id AND estado = 'ACTIVA';
    
    IF v_fecha_fin IS NULL THEN 
        SET v_dias = 0;
    ELSE 
        SET v_dias = DATEDIFF(v_fecha_fin, CURDATE());
    END IF;
    RETURN v_dias;
END;

💡 Concepto Clave: ¿Cómo funciona un LEFT JOIN con exclusión (IS NULL)?

Para un programador junior, es fundamental entender la diferencia entre conectar tablas y excluir registros:

  1. La tabla izquierda (FROM usuario): Es tu base de datos principal de personas.
  2. La tabla derecha (LEFT JOIN reservas): Es tu registro de transacciones.
  3. El pegamento (ON u.id = r.usuario_id): Es el canal que conecta a la persona con su transacción.
  4. La exclusión (WHERE r.id IS NULL): Al pedirle a la base de datos un LEFT JOIN, MySQL traerá a todos los usuarios, tengan o no reservas. Si un usuario no tiene reservas, los campos de la tabla derecha se llenarán con NULL. Al filtrar con WHERE r.id IS NULL, le estás diciendo a la base de datos: "Tráeme solo a aquellos usuarios que nunca han registrado una reserva en el sistema".

LEFT JOIN [Tabla Reservas] ON [Condición de unión por ID]
WHERE [Columna ID de la Tabla Reservas] IS [Condiciones] ;

Para un programador junior, es fundamental entender que un procedimiento almacenado no es solo una consulta, es un mini-programa que vive dentro de la base de datos:

Funciones Definidas por el Usuario (UDF) en SQL

Esta guía contiene la Plantilla Maestra de Funciones SQL diseñada a partir del análisis de las 20 funciones del ecosistema de Coworking (Membresías, Reservas, Finanzas y Accesos). Sirve como estructura base para crear, comprender y reutilizar lógica de cálculo que devuelve un único valor.

💡 Concepto Clave: ¿Qué es y por qué usar una Función (FUNCTION)?

Para un programador junior, es fundamental entender la diferencia entre un Procedimiento y una Función:

  1. Devuelve un solo valor: A diferencia de un SELECT normal o un Procedimiento, una Función siempre retorna un dato único (un número, un texto, una fecha o un booleano).
  2. Es una "Calculadora": Se usa dentro de consultas SELECT, WHERE o INSERT como si fuera una columna más (Ej: SELECT fn_dias_restantes_membresia(id) FROM usuario).
  3. No modifica datos: Una Función NUNCA puede hacer INSERT, UPDATE o DELETE. Solo lee datos (READS SQL DATA).
  4. Seguridad de Nulos: Es vital usar IFNULL() o COALESCE() dentro de la función para evitar que rompa la consulta principal devolviendo NULL.

    -- =========================================================================

Triggers (Disparadores) en SQL

Esta guía contiene la Plantilla Maestra de Triggers diseñada a partir del análisis de los 20 disparadores del ecosistema de Coworking (Membresías, Reservas, Pagos y Accesos). Sirve como estructura base para automatizar reglas de negocio, auditoría y validaciones directamente en la base de datos.

💡 Concepto Clave: ¿Qué es y por qué usar un Trigger?

Para un programador junior, un Trigger es como un "Guardia de Seguridad" o un "Asistente Automático" que vive dentro de una tabla específica:

  1. Automatización: Se ejecuta solo cuando ocurre un evento específico (INSERT, UPDATE, DELETE) en una tabla.
  2. BEFORE (Antes): Ocurre antes de que el dato se guarde. Puedes modificar el dato (SET NEW.campo = valor) o bloquear la operación (SIGNAL SQLSTATE).
  3. AFTER (Después): Ocurre después de que el dato ya se guardó. No puedes modificar el dato original, pero sí puedes usarlo para actualizar otras tablas (cascada) o guardar un registro en una tabla de logs (auditoría).
  4. NEW y OLD:
    • NEW.campo: El valor que se está intentando insertar o actualizar (disponible en INSERT y UPDATE).
    • OLD.campo: El valor que tenía el registro antes de la modificación (disponible en UPDATE y DELETE).

A continuación se presenta la Plantilla Maestra. Contiene las 3 estructuras más comunes. Copia y adapta la que necesites.

Eventos Programados (Scheduled Events) en SQL

Esta guía contiene la Plantilla Maestra de Eventos Programados diseñada a partir del análisis de los 20 eventos del ecosistema de Coworking (Membresías, Reservas, Pagos y Accesos). Sirve como estructura base para automatizar tareas de mantenimiento, reportes y notificaciones directamente en el motor de base de datos, sin depender de scripts externos.

💡 Concepto Clave: ¿Qué es y por qué usar un Evento Programado?

Para un programador junior, un Evento Programado es el equivalente a un "Cron Job" o "Tarea Programada" interno de MySQL:

  1. Automatización Nativa: Ejecuta consultas SQL automáticamente en intervalos de tiempo específicos (cada hora, cada día, cada mes) o en una fecha y hora exacta.
  2. Requisito Obligatorio: El planificador de eventos de MySQL debe estar activado en el servidor (SET GLOBAL event_scheduler = ON;). Si no lo está, los eventos se crean pero nunca se ejecutan.
  3. DO vs BEGIN ... END: Si el evento solo ejecuta una línea (ej: un UPDATE), puedes usar DO directamente. Si necesita ejecutar múltiples líneas (ej: un UPDATE y luego un INSERT en un log), debes envolver la lógica en DO BEGIN ... END.
  4. Funciones de Fecha: Son el corazón de los eventos. CURDATE() (solo fecha), NOW() (fecha y hora), DATE_ADD() (sumar tiempo) y DATE_SUB() (restar tiempo) son tus mejores amigos para filtrar datos antiguos o futuros.

Control de Acceso y Roles (RBAC) en SQL

Esta guía contiene la Plantilla Maestra de Seguridad y Roles diseñada a partir del análisis del ecosistema de Coworking. Sirve como estructura base para implementar el Principio de Menor Privilegio, garantizando que cada usuario o sistema tenga únicamente los permisos estrictamente necesarios para realizar su trabajo, protegiendo la integridad y confidencialidad de los datos.

💡 Concepto Clave: ¿Qué es RBAC y por qué usar Vistas de Seguridad?

Para un programador junior, la seguridad en bases de datos se basa en dos pilares fundamentales:

  1. RBAC (Role-Based Access Control): En lugar de dar permisos a cada usuario individualmente, creas "Roles" (ej: Recepcionista, Contador) y les asignas permisos. Luego, asignas el Rol al usuario. Si cambian las reglas, solo modificas el Rol, no a los 100 usuarios.
  2. Vistas de Seguridad (Security Views): Son como "ventanas con filtro". En lugar de darle al usuario acceso directo a la tabla usuario (donde podría ver datos de todos), le das acceso a una VIEW que filtra los datos usando la función USER(), mostrando solo su propia información.
  3. Principio de Menor Privilegio: Primero otorgas permisos amplios, luego usas REVOKE para quitar el acceso directo a las tablas sensibles y finalmente GRANT solo sobre la Vista segura.
  4. FLUSH PRIVILEGES: Obliga a MySQL a recargar las tablas de permisos en memoria. Sin esto, los cambios podrían no surtir efecto hasta que se reinicie el servidor.

🛠️ La Plantilla Maestra Comentada (Universal)

A continuación se presenta la Plantilla Maestra. Está diseñada para que puedas copiar, adaptar y escalar la seguridad de cualquier nuevo módulo o rol que se agregue al sistema.

-- ============================================================================
-- 👑 PLANTILLA MAESTRA DE SEGURIDAD Y ROLES (RBAC)
-- Diseñada para implementar el Principio de Menor Privilegio en MySQL.
-- ============================================================================

USE nombre_de_tu_base_de_datos;

-- =========================================================================
-- PASO 1: CREACIÓN DE ROLES (Los "Cargos" del sistema)
-- =========================================================================
CREATE ROLE IF NOT EXISTS 'NombreDelRol';

-- =========================================================================
-- PASO 2: ASIGNACIÓN DE PERMISOS GRANULARES AL ROL
-- Regla: Otorgar solo lo estrictamente necesario (SELECT, INSERT, UPDATE, DELETE)
-- =========================================================================
-- Permisos de escritura/lectura en tablas operativas
GRANT SELECT, INSERT, UPDATE ON nombre_base.tabla_operativa TO 'NombreDelRol';
-- Permisos de solo lectura en tablas de referencia o catálogo
GRANT SELECT ON nombre_base.tabla_catalogo TO 'NombreDelRol';

-- =========================================================================
-- PASO 3: CREACIÓN DE USUARIOS Y ASIGNACIÓN DE ROLES
-- =========================================================================
-- Crear el usuario (usar '%' permite acceso desde cualquier IP, ideal para desarrollo; en producción usar 'localhost' o IP específica)
CREATE USER IF NOT EXISTS 'nombre_usuario'@'%' IDENTIFIED BY 'ContraseñaSegura123!';

-- Asignar el rol al usuario
GRANT 'NombreDelRol' TO 'nombre_usuario'@'%';

-- Establecer el rol como predeterminado (para que el usuario no tenga que activarlo manualmente con SET ROLE)
SET DEFAULT ROLE 'NombreDelRol' TO 'nombre_usuario'@'%';

-- =========================================================================
-- PASO 4: CREACIÓN DE VISTAS DE SEGURIDAD (Opcional pero Altamente Recomendado)
-- Filtra los datos para que el usuario solo vea lo que le corresponde.
-- =========================================================================
CREATE OR REPLACE VIEW nombre_base.vista_segura_del_rol AS
SELECT columna1, columna2 
FROM nombre_base.tabla_sensible
-- El truco: Filtrar usando la función USER() que devuelve 'usuario@host'
WHERE campo_identificador = SUBSTRING_INDEX(USER(), '@', 1); 

-- =========================================================================
-- PASO 5: APLICACIÓN DEL PRINCIPIO DE MENOR PRIVILEGIO (Hardening)
-- Quitamos el acceso directo a la tabla y lo reemplazamos por acceso a la Vista.
-- =========================================================================
REVOKE ALL PRIVILEGES ON nombre_base.tabla_sensible FROM 'NombreDelRol';
GRANT SELECT ON nombre_base.vista_segura_del_rol TO 'NombreDelRol';

-- =========================================================================
-- PASO 6: APLICAR CAMBIOS INMEDIATAMENTE
-- =========================================================================
FLUSH PRIVILEGES;
